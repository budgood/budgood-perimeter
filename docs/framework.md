# 缘起工程 · Condition Engineering
## 一个以治理为先、对 Loop Engineering 的改进（超集）

> *working name——名字随你改。* 下面是被"拉出来"的框架本身：它不依赖任何具体实现（包括 manasikāra）即成立。manasikāra 在 §9 作为参考实现出现，不是框架本身。

---

### 0. 一句话

Loop Engineering 问的是："我怎么把循环闭合、让 agent 收敛？"

这套框架问的是：**"判断被允许活在哪里，以及我怎么让每一处我停止观察的地方都无法假装没看见？"**

——闭环不再是目标，它降级成这把尺子上的**一档校准设定**。

---

### 1. Loop Engineering 骑的那根轴，以及它为什么不完整

Loop Engineering = 目标 + 工具 + 自动评估 + 多 Agent + 记忆，迭代到 oracle 说"完成"。

它的隐藏前提：**判断可以、且应该被外化进循环的机器里**（eval / router / verifier）。

它的失效形态：那套机器会硬化、漂移，然后**静默地、系统性地**错——就是那个"90% 流量被误升级到最贵模型、却不报错不告警"的真实事故。它在**有硬 oracle** 的任务上优化吞吐；在其余一切任务上，它**悄悄把一次性的错误固化成结构性的错误**。

---

### 2. 这套框架放在第一位的原语：亲因 vs 增上缘（硬化刻度）

任何工件——数据、表、代码、eval、乃至循环本身——**永远只能是某个决策的增上缘（辅助条件），绝不可成为亲因（亲生因）**。亲因是当下活着的现行 / 慧。

"你把多少东西硬化进脚手架？"——这是**那个**核心设计问题，而答案要从任务里**读出来**，不是默认。

Loop Engineering 把这个旋钮**焊死在了"最大硬化"**上。这套框架的第一动作，是把旋钮拆下来重新交到现行手里。

---

### 3. 三个校准输入（什么决定旋钮的位置）

- **oracle 质量**：那个"完成"信号有多硬、多会漂？
- **不可逆性**：一次错误承诺的代价——且注意：不可逆性在**首个下游依赖**处结晶，不在写入处。一条没人读的 candidate 是无害的；一条已被下游建楼的，写下的那一刻就已不可逆。
- **可分解性（劈缝）**：能不能把混合所缘劈成"硬可验证骨架 + 悬挂的软残差"？这是把前两个轴**改写**的能力——比值不是测出来的，是设计出来的。

→ **闭环只在 oracle 硬 且 行为可逆时被许可。** 否则：只对劈出来的硬半边闭环；软残差降级成受治 candidate 挂起等人；对真正不可分的，**只给显著性，绝不闭环**。

> 中间地带（中等且会漂移的 oracle）的正确动作，**不是在尺子上找一个折中刻度**。0.5 档的旋钮不是稳健与漂移的平均，它把漂移制度化了一半、且是你最看不见的那一半。正确动作是先动刀分解，直到每个子问题都落在尺子的某一端。**curator 真正稀缺的技能是劈缝，不是 append。**

---

### 4. 落出来的两条定律

- **显著性 ≠ 控制。** 把"注意到 X"和"做 Y"切开（中断 / 处理程序那道缝）。于是你能在不动行为的前提下演化"被照亮的是什么"，并把"系统让什么变得可见"与"它据此做了什么"**分开审计**。
- **治理 = 抗坍缩。** append-only + provenance + just-in-time + fail-safe。这是**模型坍缩防御搬到了知识层**：用真实（codified）锚住合成漂移（candidate），否则系统朝自己的自信先验腐烂、丢掉分布的尾部——那些罕见但要命的判别。

---

### 5. 判断-落位必须被守住的四个 loci

记录（at rest）／ 通道（in flight）／ 值（at boundary）／ **perimeter（通道集本身）**。

绝大多数系统只治理第一个。这套框架的伸手范围：

- **沉积审计**：让通道给输出签名（provenance 加"获取通道"维）→ 会落库的流动重新在 at-rest 可被审计；
- **污点 / 自照值**：把显著性搬到值上，让值自带标签抵达最后一个 owned 边界；
- **perimeter 作为受治、受签的工件**：manifest + store 网关 + 自勘 checklist + 反身陈旧触发。

---

### 6. 顶点与诚实下限

亲因不可消灭，只可**稀释、上移**。框架的活儿是**把那个不可再分的现行的频率与表面积压到最小**，而非废除它——任何宣称废除它的系统，都是在宣称无慧而拣择，正是宪法所禁的原罪。

一个能强制自己重数 perimeter 的系统 = 自定义自身 scope = 自我立法 = 原罪。

所以终点是：让"不去看"无法假装（反身陈旧显著性）、让"去看"便宜（checklist）、让"看过了"留痕（attestation）——但**永不强制**。

**成功判据不是"永远正确"（不可达），是"永远无法悄悄相信自己正确"。**

---

### 7. 与 Loop Engineering 并排

| 维度 | Loop Engineering | 缘起工程（本框架） |
|---|---|---|
| 设计对象 | 循环 | 判断的落位 |
| 可靠性来源 | 闭合循环 | 校准的增上缘 + 无法假装的盲区 |
| 主防的失效 | 失控 / 烧钱 | 静默、系统性、伪装成完备的漂移 |
| 无 oracle、高不可逆的活 | 禁区 / 脆弱 | 原生（只给显著性） |
| 对判断的态度 | 外化进机器 | 留在现行 |
| 度量诚实 | done 信号 | 新鲜度 / 随结清而非摄入 |

---

### 8. 关系：它不是对手，是超集

Loop Engineering = 这套框架在**某一档旋钮设定**（硬 oracle、可逆、已完全分解）下的特例。

所以它不在"比 Loop Engineering 更好"这个竞品意义上成立；它是那个**告诉你"循环什么时候是对的设定、什么时候是渎职"的上位理论**。这才是它"改进" Loop Engineering 的精确含义：它补上了 Loop Engineering 缺失的**校准理论**，并拒绝 Loop Engineering 的普适主义。

> 顺着这条线的源头收口：这恰恰是"以数据为出发点"那个直觉的成熟形态。
> **Loop Engineering →（真正上位的词）context engineering → 这套"长出了宪法、校准理论和反身 perimeter 的 context engineering"。**

---

### 9. manasikāra：参考实现，不是框架本身

manasikāra 是这套框架在**"无 oracle / 高不可逆 / 部分可分解"那个最硬的角**上的实例化：只给显著性、append-only 治理、判断留在现行、为自己的盲区做 perimeter 自勘。

它是这套框架最难那一角的**参考构建**——可照抄、可参考——但它**不是框架**。框架是上面这把尺子；manasikāra 是尺子在一个极端读数处凝成的一件实物。
